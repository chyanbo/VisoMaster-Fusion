### BUG-05 🔴 — Division by zero in `set_gpu_memory_progressbar_value`
- **File:** `app/ui/widgets/actions/common_actions.py`
- **Location:** `set_gpu_memory_progressbar_value()` ~line 427
- **Description:**
  ```python

  ```
  If `memory_total` is `0` (no GPU detected, or CPU-only mode), a `ZeroDivisionError` is raised, crashing the application silently.

### BUG-06 🔴 — Division by zero in slider `paintEvent` (custom video seek slider)
- **File:** `app/ui/widgets/actions/video_control_actions.py`
- **Location:** `set_up_video_seek_slider()` → inner `paintEvent()` ~line 84–86
- **Description:**
  ```python
  normalized_value = (self.value() - self.minimum()) / (self.maximum() - self.minimum())
  ```
  If `maximum == minimum` (e.g., when no video is loaded and slider is at `[0, 0]`), this is a division by zero. The same applies to marker position calculation (~line 118).

### BUG-07 🔴 — `time.sleep(0.5)` blocks the main/GUI thread
- **File:** `app/ui/widgets/actions/list_view_actions.py`
- **Location:** `clear_stop_loading_target_media()` ~line 249 and `clear_stop_loading_input_media()` ~line 338
- **Description:**
  ```python
  time.sleep(0.5)
  ```
  `time.sleep` is called from the main GUI thread while clearing the loader worker. This freezes the entire UI for 500 ms on every folder change or workspace load. Combined with `worker.wait()` (see BUG-08), it can freeze the UI for even longer.
- **Fix:** Replace with `QCoreApplication.processEvents()` after setting `_running = False`, or use `QThread.wait()` with a timeout and skip the sleep entirely.

### BUG-08 🔴 — `TargetMediaLoaderWorker.stop()` calls `self.wait()` (potential deadlock)
- **File:** `app/ui/widgets/ui_workers.py`
- **Location:** `TargetMediaLoaderWorker.stop()` ~line 157 and `InputFacesLoaderWorker.stop()` ~line 292
- **Description:**
  ```python
  def stop(self):
      self._running = False
      self.wait()
  ```
  `self.wait()` is a blocking call. If `stop()` is called from the main thread while the worker is blocked on a slow I/O call (e.g., opening a large video file), the main thread stalls until the worker finishes its current operation. When combined with `clear_stop_loading_target_media()`'s `time.sleep(0.5)`, the UI can freeze for over 1 second.
- **Fix:** Use `self.wait(timeout_ms)` with a reasonable timeout, or signal the worker via `quit()` and use a non-blocking `QTimer`-based retry for the list clear.

### BUG-12 🔴 — `rename_preset`: `_ctl.json` rename can fail after main preset was renamed
- **File:** `app/ui/widgets/actions/preset_actions.py`
- **Location:** `rename_preset()` ~lines 56–57
- **Description:**

  If `old_path.rename(new_path)` succeeds but `old_path_ctl.rename(new_path_ctl)` fails (e.g., `_ctl.json` does not exist), the preset files are left in an inconsistent state (main file renamed, control file not renamed). The `except` block cannot undo the first rename.

### BUG-13 🔴 — `handle_reference_kv_file_change`: `model_lock` held during slow `torch.load()`
- **File:** `app/ui/main_ui.py`
- **Location:** `handle_reference_kv_file_change()` ~line 619
- **Description:**
  ```python
  with self.models_processor.model_lock:
      ...
      kv_payload = torch.load(kv_file_path, ...)
  ```
  The `model_lock` is held while performing a potentially slow file I/O operation (`torch.load`). This blocks any other thread (e.g., the frame worker) from accessing the model lock, potentially causing stuttering or hangs during loading. Additionally, `self.model_loading_signal.emit()` is called inside the lock, which could cause deadlocks if the connected slot tries to acquire the same lock.
- **Fix:** Load the file outside the lock, then acquire the lock only to update the shared `current_kv_tensors_map` attribute.

### BUG-14 🟠 — `FilterWorker` reads UI widgets from a background thread (thread-safety violation)
- **File:** `app/ui/widgets/ui_workers.py`
- **Location:** `FilterWorker.filter_target_videos()`, `filter_input_faces()`, `filter_merged_embeddings()` ~lines 337–380
- **Description:**
  ```python
  search_text = main_window.targetVideosSearchBox.text().lower()  # UI widget read from thread!
  for i in range(main_window.targetVideosList.count()):           # QListWidget read from thread!
      item = main_window.targetVideosList.item(i)
      item_widget = main_window.targetVideosList.itemWidget(item)
      if ... item_widget.media_path.lower() ...                  # Widget property from thread!
  ```
  Qt explicitly forbids accessing GUI objects from non-GUI threads. This pattern can cause crashes, data races, or silent data corruption.
- **Fix:** Capture the needed data (search text, list of media paths/names with their list indices) in the main thread *before* starting the FilterWorker, pass it as constructor arguments, and have the worker operate only on the captured plain data.

## 3. Performance Issues

### PERF-01 🟡 — `set_control_widgets_values` rebuilds `all_widget_options` dict on every call
- **File:** `app/ui/widgets/actions/common_actions.py`
- **Location:** `set_control_widgets_values()` ~lines 574–582
- **Description:**
  ```python
  all_widget_options = {}
  for layout_data_source in [SETTINGS_LAYOUT_DATA, COMMON_LAYOUT_DATA]:
      for group_name, group_data in layout_data_source.items():
          for widget_key, widget_data in group_data.items():
              all_widget_options[widget_key] = widget_data
  ```
  This dictionary is rebuilt from scratch every time `set_control_widgets_values` is called (which happens on every workspace load). Since `SETTINGS_LAYOUT_DATA` and `COMMON_LAYOUT_DATA` are module-level constants, this computation is wasteful.
- **Fix:** Compute this merged dictionary once at module load time as a module-level constant:
  ```python
  _ALL_WIDGET_OPTIONS = {k: v for d in [SETTINGS_LAYOUT_DATA, COMMON_LAYOUT_DATA]
                         for grp in d.values() for k, v in grp.items()}
  ```

### PERF-04 🟡 — `paintEvent` sorts markers set on every repaint

- **Description:**
  ```python
  for value in sorted(self.markers):
  ```
  `sorted()` creates a new sorted list from the markers set on every single paint event. The seek slider is repainted frequently (on every frame). For large numbers of markers, this is wasteful.
- **Fix:** Maintain a sorted list in parallel with the set, or sort only when markers are added/removed (cached via a dirty flag).

### PERF-05 🟡 — `refresh_frame` called redundantly during bulk parameter updates
- **File:** `app/ui/widgets/actions/common_actions.py`
- **Location:** `update_parameter()` ~line 175 and `set_widgets_values_using_face_id_parameters()` ~line 530–561
- **Description:** When switching faces or loading a workspace, `set_widgets_values_using_face_id_parameters` sets `widget.enable_refresh_frame = False` during bulk updates, which correctly suppresses individual `refresh_frame` calls. However, the final `widget.enable_refresh_frame = True` is set *before* the single refresh at the end, meaning if anything else triggers a refresh during the loop, it could fire. More critically, `update_parameter` still calls `refresh_frame` for each parameter unless the caller explicitly disables it.
- **Fix:** Add a `main_window.batch_update_in_progress` flag that suppresses all `refresh_frame` calls, and explicitly call `refresh_frame` once after the batch is complete.

### PERF-06 🟡 — `clear_stop_loading_target_media` and `clear_stop_loading_input_media` clear UI after sleep
- **File:** `app/ui/widgets/actions/list_view_actions.py`
- **Location:** `clear_stop_loading_target_media()` ~lines 243–250
- **Description:** After `stop()` and `terminate()` are called, a `time.sleep(0.5)` is used as a workaround to "wait for the thread to stop" before clearing the list. This is a race condition workaround using an arbitrary timeout.
- **Fix:** Use `QThread.wait()` with a timeout (e.g., 1000 ms), then clear only if the thread has actually stopped. If the thread hasn't stopped within the timeout, force-terminate via `QThread.terminate()` and then clear.

## 4. Thread-Safety Issues

### THREAD-01 🔴 — `FilterWorker` reads Qt widgets from a background thread
- **See BUG-14 above for full details.**
- **Summary:** All three filter methods access `QListWidget`, `QLineEdit`, and custom widget properties directly from a `QThread` worker. This violates Qt's thread-safety rules.

---

### THREAD-02 🟠 — `TargetMediaLoaderWorker` accesses `ThumbnailManager` from a worker thread
- **File:** `app/ui/widgets/ui_workers.py` and `app/ui/widgets/actions/common_actions.py`
- **Location:** `load_videos_and_images_from_folder()` → calls `extract_frame_as_pixmap()` → accesses `main_window.thumbnail_manager`
- **Description:** `ThumbnailManager` is accessed concurrently from multiple worker threads (if multiple loaders run). If `ThumbnailManager` is not internally thread-safe (using file I/O without locking), this can cause race conditions.
- **Fix:** Add a `threading.Lock` inside `ThumbnailManager` protecting file existence checks and writes.

- **See BUG-16 above for full details.**

### THREAD-04 🟠 — `InputFacesLoaderWorker` runs GPU inference in a background thread without explicit GPU context management
- **File:** `app/ui/widgets/ui_workers.py`
- **Location:** `InputFacesLoaderWorker.load_faces()` ~lines 228–264
- **Description:** The worker calls `models_processor.run_detect()` and `run_recognize_direct()` from a background thread. These use CUDA. While the `model_lock` in `ModelsProcessor` likely protects model state, CUDA contexts and streams may not be properly managed for multi-threaded access if the workers and the main processing thread share a CUDA stream.
- **Fix:** Ensure the models processor uses CUDA stream synchronization or that background workers use a separate CUDA stream context.
